# MemsiInterface — Smoke Tests

חמשת התרחישים הידניים שצריכים לעבור אחרי כל ספרינט שמסתיים. מטרה: לתפוס regressions לפני tag.

הכוונה לבדיקות-ידיים (manual) שצריכות לקחת ביחד ~15 דקות. אין אוטומציה כרגע — אם ייכתבו tests אוטומטיים בעתיד, הם יוסיפו לרשימה זו, לא יחליפו אותה.

## הכנה

לפני כל ריצת smoke:

1. ודא שיש `.env` עם:
   - `PRIORITY_AUTH_HEADER`
   - DB credentials (DATABASE_URL או DB_HOST/PORT/NAME/USER/PASSWORD)
2. ודא ש-postgres רץ ונגיש: `psql ... -c "SELECT 1"`
3. ודא שהמיגרציות אקטואליות: `python migrate.py --status`. אם יש pending — `python migrate.py`.

## תרחיש 1 — Startup עם DB מחובר

**מה עושים:** `python gui_app.py`

**מה אמור לקרות:**
- חלון נפתח תוך < 5 שניות.
- ה-status bar אינו מציג שגיאת-DB.
- כל הלשוניות נטענות (לא קורסות בלחיצה).
- ה-log ב-`memsi.log` מכיל הודעת startup ולא ERROR.

**מה אסור לקרות:**
- QMessageBox קופץ.
- האפליקציה תקועה ללא תגובה לאחר startup.

**מה אסור לקרות:**
- Traceback ב-UI.
- האפליקציה תקועה.

## תרחיש 3 — Backtest 6-חודשים על קטגוריה אחת

**מה עושים:**
1. לשונית "תחזיות".
2. בחירת קטגוריה "מזוודות נוקשות גדולות" (או דומה שיש לה היסטוריה ארוכה).
3. בחירת horizon = 6 חודשים.
4. לחיצה על "הרץ תחזית".

**מה אמור לקרות:**
- 3 מודלים רצים (ARIMA, Prophet, XGBoost).
- progress mostly אינו תקוע על מודל בודד יותר מ-2 דקות.
- חלון תוצאות מציג גרפים, MAE/RMSE, ו-Newsvendor.
- ה-MAE/RMSE/MAPE שונים ממה שהיה לפני A1 (אם החודש האחרון של ה-history נופל על אירוע בעבר).
- ב-DB: `SELECT run_id, is_legacy FROM forecast_runs ORDER BY ran_at DESC LIMIT 1` מחזיר `is_legacy = false`.

**מה אסור לקרות:**
- כשל באחד המודלים מעצור את האחרים (לאחר A3 — אבל גם ב-A1 חשוב לוודא שלפחות 2 מ-3 שורדים).
- היעדר Newsvendor בפלט.

## תרחיש 4 — עדכון מחיר ב-UpdatesTab

**מה עושים:**
1. לשונית "עדכונים".
2. בחירת לקוח קיים (למשל "אל-על").
3. שינוי מחיר תיקון לדגם ספציפי.
4. לחיצה על "שמור".

**מה אמור לקרות:**
- הודעת אישור.
- ה-DB עודכן: `SELECT * FROM customer_repair_prices WHERE customer_name = 'אל-על' AND ...`
- אם פותחים את הלשונית מחדש, המחיר החדש מוצג.

**מה אסור לקרות:**
- Traceback או QMessageBox-של-שגיאה.
- מצב שבו ה-UI מציג אחד והDB אחר (commit שלא קרה).

## תרחיש 5 — InventoryTab עם נתונים גדולים

**מה עושים:**
1. לשונית "מעקב מלאי".
2. בחירת "כל המחסנים" (או רוב המחסנים) ו-"כל הספקים".
3. לחיצה על "טען מלאי".

**מה אמור לקרות:**
- progress bar מתקדם.

- ניתן לגלול בלי קפיאות מורגשות.

**מה אסור לקרות:**
- האפליקציה קופאת ללא תגובה.
- שגיאת out-of-memory.

## עברתי את כל ה-5 — מה עכשיו?

1. `git add -A && git commit -m "..."` עם הודעה ברורה.
2. אם הספרינט הסתיים: `git tag v0.X.0` והעלאה ל-remote (אם יש).
3. עדכון `MEMORY.md` או log פנימי במצב הספרינט.

## לא עברתי — מה עכשיו?

1. אל תעשה tag.
2. נסה לזהות אם זה regression חדש או באג קיים.
3. אם regression — חזור-לאחור (`git revert` או `git reset --hard <tag>`) או תקן מיד.
4. אם באג קיים — פתח issue, תזכור לטפל בספרינט הבא.
